# 017 — Per-place order expiration window (default 3 days)

**Status:** done
**Type:** feature / config
**Scope:** small (~10 files)
**Depends on:** none

## Problem

Order expiration is hardcoded at 7 days for every place — `OrderService::RESERVATION_DAYS = 7` (`src/Service/OrderService.php:27`) is used in `createOrder()` at line 92 to set `Order::$expiresAt`. The customer sees this on the payment page (`templates/public/order_payment.html.twig:99`: *"Objednávka vyprší {{ order.expiresAt|date('d.m.Y H:i') }}"*) and in the order-confirmation e-mail.

Different places need different windows. 7 days is too generous for high-turnover locations; the default should be 3.

## Goal

The expiration window is a per-place setting, configurable on the place edit/create form, defaulting to 3 days. Existing places are backfilled to 3 by the migration. New orders use the place's value; existing in-flight orders are unaffected (their `expiresAt` is already persisted on `Order`).

## Context (current state)

- **Where it's applied:** `src/Service/OrderService.php:92` — `expiresAt: $now->modify('+'.self::RESERVATION_DAYS.' days')` inside `createOrder(...)`. The method already receives `Place $place` (line 48), so plumbing the value through is just a property read.
- **Place entity:** `src/Entity/Place.php` — uses PHP 8.4 property hooks. Note: a dormant `int $daysInAdvance = 0` field exists at line 32 (with `updateDaysInAdvance` at line 109) but is **unreferenced anywhere outside the entity and its unit test**. Do **not** repurpose it (semantics differ — "booking lead time" vs. "order expiration"). Leave it alone; cleanup is a separate task.
- **Place form pair:** `src/Form/PlaceFormData.php` and `src/Form/PlaceFormType.php` — straight FormData + FormType per CLAUDE.md.
- **Place commands:** `src/Command/CreatePlaceCommand.php`, `src/Command/UpdatePlaceCommand.php`, with handlers `CreatePlaceHandler.php` / `UpdatePlaceHandler.php`. Handlers call entity behavior methods (no setters).
- **Templates render the form manually** (label + `form_widget` + `form_errors` per row): `templates/portal/place/edit.html.twig` and `templates/portal/place/create.html.twig`. Both must get the new row — `form_end()` will *not* auto-render unrendered fields here because every other field is rendered manually too, but better to render explicitly so we never depend on Symfony's "rest" behavior.
- **Display surfaces (no change required, they read `order.expiresAt` which is already a stored timestamp):** `templates/public/order_payment.html.twig:99`, `templates/email/order_confirmation.html.twig`, the various order detail templates.
- **Tests touching `OrderService::createOrder`:** `tests/Integration/Service/OrderWorkflowTest.php`, `tests/Integration/Command/ChargeRecurringPaymentHandlerTest.php`, `tests/Unit/Command/CustomerSignOnboardingHandlerTest.php`, `tests/Unit/Command/AdminMigrateCustomerHandlerTest.php`. They mostly use fixture places, so backfill default of 3 carries them; spot-check assertions that compute an expected `expiresAt` from the current `+7 days`.
- **Place tests:** `tests/Unit/Entity/PlaceTest.php` — has a constructor-defaults assertion at line 133 (`$this->assertSame(0, $place->daysInAdvance);`). Add a parallel assertion for the new field's default.

## Requirements

### 1. Place entity — new property + behavior method

`src/Entity/Place.php`

Add a new property near the existing `daysInAdvance` (keep both — they are unrelated). Default **3** so freshly constructed `Place` objects without explicit value match the new business rule.

```php
#[ORM\Column]
public private(set) int $orderExpirationDays = 3;
```

And the corresponding behavior method (mirrors `updateDaysInAdvance`'s shape):

```php
public function updateOrderExpirationDays(int $orderExpirationDays, \DateTimeImmutable $now): void
{
    $this->orderExpirationDays = $orderExpirationDays;
    $this->updatedAt = $now;
}
```

No constructor argument — this is a managed setting, not identity-time data.

### 2. Migration

Generate via `docker compose exec web bin/console make:migration` (CLAUDE.md mandates this; do **not** handwrite SQL). Verify the resulting migration:

- Adds `order_expiration_days INT NOT NULL` to `place`.
- Includes a backfill: `UPDATE place SET order_expiration_days = 3` (the autogenerated `ALTER TABLE … ADD COLUMN … NOT NULL` will need a default during the add-and-backfill step; if Doctrine generates it without a default for an existing-row table, edit the migration body so it adds with default 3, backfills, then drops the default — standard Postgres pattern). The Doctrine schema is column-without-default (`#[ORM\Column]` no `options: ['default' => 3]`), so the runtime default lives in PHP, not SQL.

After generation, run `docker compose exec web composer db:reset` and `bin/console doctrine:schema:validate` — both must be clean.

### 3. Form data + form type

`src/Form/PlaceFormData.php`

Add the field with validation. Range: 1–30. Defaulted to 3 so the create form pre-fills sensibly.

```php
#[Assert\NotNull(message: 'Zadejte počet dní platnosti objednávky')]
#[Assert\Range(min: 1, max: 30, notInRangeMessage: 'Zadejte hodnotu mezi {{ min }} a {{ max }} dny')]
public int $orderExpirationDays = 3;
```

Update `PlaceFormData::fromPlace()` to copy the value:

```php
$formData->orderExpirationDays = $place->orderExpirationDays;
```

`src/Form/PlaceFormType.php` — add the field. Use `IntegerType` with explicit min/max attributes for nice browser validation:

```php
$builder->add('orderExpirationDays', IntegerType::class, [
    'label' => 'Doba platnosti objednávky (dny)',
    'help' => 'Po této době nezaplacená objednávka automaticky vyprší. Výchozí 3 dny.',
    'attr' => [

        'max' => 30,

    ],

```

Place the field somewhere logical — suggested: under `description`, before the "Soubory" `<hr>`.


Default to 3 to keep existing call sites compiling without forcing every test fixture to specify it (none currently exists in the codebase outside the controllers, but defensive default = less churn).

Be sure to call it unconditionally (no `if` guard — the value is always set).

`src/Command/UpdatePlaceHandler.php` — add `$place->updateOrderExpirationDays($command->orderExpirationDays, $now);` alongside the existing update calls.

## Acceptance

- `docker compose exec web composer quality` is green.
- Place edit page (e.g. `/portal/places/{id}/edit`) shows a "Doba platnosti objednávky (dny)" number input pre-filled with the place's current value (3 for backfilled places). Saving 5 persists 5.
- Place create page (`/portal/places/create`) shows the same field pre-filled with 3. Submitting creates a place with the chosen value.
- Validation: submitting 0 or 31 fails with the range error message; non-integer fails with the field-type error.
- New order placed against a place with `orderExpirationDays = 5` shows `expiresAt = createdAt + 5 days` on `/objednavka/{id}/platba`.
- Existing in-flight orders (created before this deploy) keep their original `expiresAt` — no retroactive change. Verify by inspecting an order created during fixtures load before/after the migration.
- `bin/console doctrine:schema:validate` is clean.
- `bin/console doctrine:migrations:status` reports up-to-date.

## Out of scope

- Removing the dormant `daysInAdvance` field/method on `Place`. It's unused outside the entity and its single unit-test assertion, but cleanup is a separate concern and the user did not request it.
- Cron / scheduled-task tuning around order expiration (`SendExpirationRemindersCommand` etc.). The reminder schedule is independent of the per-place window; if a follow-up wants reminders to scale with the window, that's a separate spec.
- Customer-facing copy changes elsewhere (order confirmation e-mail, order detail pages). They already render `order.expiresAt` directly — no template change needed.
- Per-place override at order-creation time (admin overriding for a one-off order). Not requested.
- Showing the configured window anywhere on the public place page or storage browse views.
