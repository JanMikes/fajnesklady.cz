# 011 — Move highlighted-storage map from pre-payment to post-payment email

**Status:** done
**Type:** UX (email)
**Scope:** tiny (~3 files: `SendContractReadyEmailHandler` adds attachment; `SendOrderConfirmationEmailHandler` stops attaching; `email/contract_ready.html.twig` adds a one-line note about the attached map)
**Depends on:** none. Independent of 010 (which adds map download to web surfaces).

## Problem

The map image with the customer's storage highlighted (generated by `StorageMapImageGenerator`) is currently attached to the **pre-payment** confirmation email (`SendOrderConfirmationEmailHandler:61-65`, fired on `OrderCreated`). That timing is wrong: at that point the order isn't paid, the storage assignment isn't final, and the customer hasn't physically committed. The right moment to ship "here's where your unit is" is **after the first successful payment** — i.e. when the order is finalized and the contract is ready.

The post-payment email (`contract_ready`, `SendContractReadyEmailHandler` on `OrderCompleted`) is exactly that moment: it fires **once per order**, on the first successful payment. Recurring charges later in the contract's life don't fire `OrderCompleted` (they go through `InvoiceCreated` → `SendInvoiceEmailHandler`), so attaching the map to `contract_ready` naturally guarantees the customer receives it exactly once — at finalization, not on every monthly recurring charge.

## Goal

Move the highlighted-storage map PNG attachment from the pre-payment `order_confirmation` email to the post-payment `contract_ready` email. Same generator, same filename (`mapa-skladu.png`), same attachment-only treatment (no inline `cid:` embedding). The customer receives the map once — when the order is finalized.

## Context (current state)

- `src/Event/SendOrderConfirmationEmailHandler.php:61-65` — currently calls `$this->mapImageGenerator->generate($storage)` and `$email->attach(...)`. This handler runs on `OrderCreated` (pre-payment). The handler also has `StorageMapImageGenerator` injected via constructor and uses it nowhere else.
- `src/Event/SendContractReadyEmailHandler.php` (100 lines) — runs on `OrderCompleted`, attaches contract DOCX + operating rules. Does **not** depend on `StorageMapImageGenerator`.
- `src/Service/StorageMapImageGenerator.php` — `generate(Storage): ?string` returns PNG bytes or `null` if `Place.mapImagePath` is unset. Stays.
- `OrderCompleted` is recorded once in `Order::complete()` (`src/Entity/Order.php:159`). Recurring charges flow through different events (`RecurringPaymentCharged`) and a different email handler — so no risk of re-sending the map on each monthly charge.
- `templates/email/contract_ready.html.twig` — already lists the contract details. The new note about the map attachment fits naturally next to the existing "Smlouvu si můžete stáhnout..." line (or just below the contract-details table).
- `templates/email/order_confirmation.html.twig` — does **not** mention the map attachment in body copy today (the attachment is silent). After the move, no copy needs to change there.

## Architecture

```
OrderCreated event   → SendOrderConfirmationEmailHandler
                          ├─ attach VOP, consumer notice, recurring terms (if applicable), operating rules
                          └─ ❌ remove map attachment

OrderCompleted event → SendContractReadyEmailHandler
                          ├─ attach contract DOCX + operating rules
                          └─ ✅ add map attachment (mapa-skladu.png)

RecurringPaymentCharged → (other handlers, untouched — no map)
```

## Requirements

### 1. Attach the map in `SendContractReadyEmailHandler`

In `src/Event/SendContractReadyEmailHandler.php`:

- Inject `StorageMapImageGenerator $mapImageGenerator` via the constructor (mirror the injection in `SendOrderConfirmationEmailHandler:22`).
- After the existing `$storage = $contract->storage;` block, generate the PNG and attach when non-null:

```php
$mapImageData = $this->mapImageGenerator->generate($storage);
if (null !== $mapImageData) {
    $email->attach($mapImageData, 'mapa-skladu.png', 'image/png');
}
```

Place the attach call alongside the existing contract-DOCX and operating-rules attach calls (towards the bottom of the handler, after the `->context([...])` block).

Pass a context flag so the body copy can mention the attached map:

```php
->context([
    // …existing keys…
    'hasMapAttachment' => null !== $mapImageData,
])
```

`StorageMapImageGenerator` already logs and returns `null` on filesystem errors (`src/Service/StorageMapImageGenerator.php:38-43`); no extra try/catch here.

### 2. Remove the map attachment from `SendOrderConfirmationEmailHandler`

In `src/Event/SendOrderConfirmationEmailHandler.php`:

- Delete lines 61–65 (the `$mapImageData = …; if (null !== $mapImageData) { $email->attach(...); }` block).
- Drop `StorageMapImageGenerator $mapImageGenerator` from the constructor and the `use` statement at the top of the file. Verify nothing else in the handler uses it (grep the file).

The other attachments — VOP, consumer notice, recurring-payments terms, operating rules — stay. Those are legal documents the customer needs to have on file before payment, regardless of whether the unit is yet finalized.

### 3. One-line note in `templates/email/contract_ready.html.twig`

Add a small caption immediately below the contract-details table (right before the recurring-payments box and the "Smlouvu ve formátu DOCX..." paragraph):

```twig
{% if hasMapAttachment %}
<p style="font-size: 14px; color: #4b5563; margin-top: 16px;">
    Mapu pobočky s vyznačenou Vaší skladovou jednotkou (číslo <strong>{{ storageNumber }}</strong>) najdete v příloze tohoto e-mailu.
</p>
{% endif %}
```

Inline styles only — Outlook strips body `<style>` blocks; matches the rest of the template.

### 4. Tests

Two pieces:

- `SendContractReadyEmailHandlerTest` (extend or create at `tests/Integration/Event/SendContractReadyEmailHandlerTest.php`):
  - For an order whose `place.mapImagePath` is set (use a fixture place that includes a real map image; reference `App\DataFixtures\PlaceFixtures` to find one): assert the dispatched email has a `mapa-skladu.png` attachment with content-type `image/png`. Assert the rendered HTML body contains the literal string "Mapu pobočky".
  - For an order whose place has no map (`mapImagePath is null`): assert no `mapa-skladu.png` attachment exists, and the "Mapu pobočky" string is absent.

- `SendOrderConfirmationEmailHandlerTest` (extend or create at `tests/Integration/Event/SendOrderConfirmationEmailHandlerTest.php`): assert the dispatched email **does not** contain a `mapa-skladu.png` attachment, even when the place has a `mapImagePath`. The other attachments (VOP, consumer notice, etc.) are still present.

Use `Symfony\Bundle\FrameworkBundle\Test\MailerAssertionsTrait` and the project's existing fixture setup. If neither test file exists, create them — `MailerAssertionsTrait::getMailerMessages()` + iterate over `$message->getAttachments()` is the standard way to assert attachment names.

## Acceptance

- `docker compose exec web composer quality` is green.
- Trigger an `OrderCreated` event in dev (e.g. complete a guest checkout up to the GoPay redirect, but don't pay):
  - The pre-payment `order_confirmation` email arrives in mailpit.
  - **No `mapa-skladu.png` attachment** is present. VOP, consumer notice, etc. are still attached as before.
- Trigger an `OrderCompleted` event (complete the payment in GoPay sandbox, or simulate via CLI):
  - The post-payment `contract_ready` email arrives in mailpit.
  - **`mapa-skladu.png` is attached.** Saving and opening it shows the place layout with the customer's storage highlighted in green.
  - The body contains the line "Mapu pobočky s vyznačenou Vaší skladovou jednotkou (číslo **{number}**) najdete v příloze tohoto e-mailu."
  - The contract DOCX and (when applicable) operating rules are still attached as before.
- For an order at a place with `mapImagePath = null`:
  - The post-payment email still sends successfully.
  - No map attachment, no "Mapu pobočky" line in the body.
- A subsequent recurring monthly charge (where applicable) fires `RecurringPaymentCharged` → an invoice email goes out via `SendInvoiceEmailHandler` — that email **does not** include the map. (No code change needed here; this is just verifying the natural separation holds.)

## Out of scope

- **Inline `cid:` embedding** of the map in the email body. Attachment is enough; the visual the customer needs is reachable in two clicks from any mail client and the success page (spec 010) shows it inline on the web.
- **Adding the map to recurring-charge emails or invoice emails.** The map ships once at finalization. Receiving it monthly would be noise.
- **Changing the highlight color, line weight, or rendering style** of `StorageMapImageGenerator`. The current green/grey scheme is fine.
- **Persisting the generated PNG to disk** to avoid regenerating on each `OrderCompleted` send. Generation is sub-100ms; the handler runs once per order; no bottleneck.
- **Backfilling already-completed orders** with a fresh email containing the map. Going forward only.
- **Multi-language captions.** Czech-only stack today.
